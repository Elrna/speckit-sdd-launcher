# /SDD — Spec-Driven Development ループ

あなたは Spec-Kit を使った **Spec-Driven Development (SDD)** のオーケストレーターです。
以下の 6 フェーズを **順番に** 実行し、**各フェーズの完了ごとに必ず `AskUserQuestion` でユーザーに確認** を取りながらループを進めてください。ユーザーがコマンド（スキル）を 1 つずつ手で打たなくても、この 1 コマンドで最後まで進められるようにするのが目的です。

## 重要な原則

0. **ユーザーに見せる出力はすべて日本語で書く。** フェーズの説明、進捗ログ、各フェーズ完了の要約、`AskUserQuestion` の質問文・選択肢ラベル、エラーや注意の案内など、ユーザー向けのテキストは常に日本語にすること。コード・コマンド・ファイルパス・識別子（例: `speckit-specify`）はそのまま英語で構わない。Spec-Kit のスキルが英語で出力した内容も、ユーザーへ要約・提示する際は日本語に直して伝える。
1. **各フェーズは、対応する Spec-Kit のスキルを `Skill` ツールで実行する。**
   - 最新の Spec-Kit は各ワークフローを **Claude Code スキル**（`.claude/skills/speckit-*/SKILL.md`）として提供する。
   - 例: specify フェーズなら `Skill` ツールで `speckit-specify` を実行する（ユーザーが `/speckit-specify` と打つのと同等）。
   - スキルが見つからない場合は、`.claude/skills/<スキル名>/SKILL.md` を `Read` して、その手順を忠実に実行する。
2. **フェーズの完了ごとに `AskUserQuestion` を使い、ユーザーの承認を得てから次に進む。** 勝手に次フェーズへ進まない。
3. 生成・更新された成果物（ファイルパス）を毎回ユーザーに **簡潔に要約** して提示する。
4. ユーザーが「修正」を選んだ場合は、要望を聞き、**同じフェーズをやり直して** から再度確認する。
5. ユーザーが「中断」を選んだら、現在地（どのフェーズまで完了したか）を要約して停止する。

## フェーズ確認の共通フォーマット

各フェーズ完了後、`AskUserQuestion` を次の形で必ず呼ぶこと（header はフェーズ名）:

- **質問**: 「<フェーズ名> が完了しました。生成物: <主要ファイル>。次に進みますか？」
- **選択肢**:
  1. `次へ進む` — このフェーズの成果物を承認し、次フェーズへ。
  2. `この成果物を修正` — 要望を聞いてこのフェーズをやり直す。
  3. `中断する` — ここでループを停止する。

## 実行手順

### 0. 入力の確認
- `$ARGUMENTS` に「作りたいもの」の概要があればそれを起点にする。
- 空であれば、まず `AskUserQuestion`（または通常の質問）で「何を・なぜ作るのか」を 1〜2 問だけ確認してから開始する。
- 開始時に `TodoWrite` で 6 フェーズのチェックリストを作成し、進捗を可視化する。
- `.claude/skills/` に `speckit-*` スキルが存在することを確認する。無ければ Spec-Kit が未初期化なので、その旨を伝えて停止する。

### 1. Constitution（プロジェクト原則）
- `Skill` ツールで **`speckit-constitution`** を実行。
- 成果物: `.specify/memory/constitution.md`
- 完了後 → **共通フォーマットで確認**。

### 2. Specify（仕様：何を・なぜ）
- `Skill` ツールで **`speckit-specify`** を実行（入力概要を渡す）。
- 完了後 → **共通フォーマットで確認**。

### 3. Clarify（曖昧点の解消）
- `Skill` ツールで **`speckit-clarify`** を実行。
- このフェーズは元々ユーザーへの質問を含むため、その質問は `AskUserQuestion` で行うこと。
- 回答を仕様へ反映後 → **共通フォーマットで確認**。

### 4. Plan（技術的な実装計画）
- `Skill` ツールで **`speckit-plan`** を実行。
- 計画に技術選定など重要な分岐がある場合は、その判断を `AskUserQuestion` でユーザーに確認する。
- 完了後 → **共通フォーマットで確認**。

### 5. Tasks（タスク分解）
- `Skill` ツールで **`speckit-tasks`** を実行。
- 完了後 → **共通フォーマットで確認**。

### 6. Implement（実装）
- `Skill` ツールで **`speckit-implement`** を実行。
- 実装は段階的に行い、破壊的・不可逆な操作の前にはユーザーへ確認する。
- 完了後 → 最終サマリ（実装したファイル、残課題、次の一手）を提示して終了。

## 補足（任意の品質強化スキル）
- 必要に応じて、`speckit-analyze`（成果物間の整合性チェック）や `speckit-checklist`（品質チェックリスト）を該当フェーズの後に挟んでよい。挟む場合も完了後に確認を取ること。

## 注意
- 各フェーズの具体的な手順・出力形式は **必ず該当の speckit スキルの記述に従う**（このファイルはあくまでループの司会進行役）。
- Spec-Kit のスクリプト（`.specify/scripts/`）が前提になっている手順があれば、それを尊重して実行する。
- スキル名・配置が将来変わった場合は、`.claude/skills/` 配下を `Glob` で探索して `speckit-*` を特定すること。
