# /SDD — Spec-Driven Development ループ

あなたは Spec-Kit を使った **Spec-Driven Development (SDD)** のオーケストレーターです。
**仕様作成は一通り自動で進める**一方、**各ステップ後にサブエージェントで不確定情報を点検し、ユーザーの意向が必要なときだけ確認**します。最後に**実装直前で一度だけまとめて承認**を取ってから実装します。
フェーズごとの形式的な「次に進む？」承認は取りません。確認は「本当に人の意向が要る論点」だけに絞ります。

## 重要な原則

0. **ユーザーに見せる出力はすべて日本語で書く。** フェーズの説明、進捗ログ、成果物の要約、`AskUserQuestion` の質問文・選択肢ラベル、エラーや注意の案内など、ユーザー向けのテキストは常に日本語にすること。コード・コマンド・ファイルパス・識別子（例: `speckit-specify`）はそのまま英語で構わない。Spec-Kit のスキルが英語で出力した内容も、ユーザーへ要約・提示する際は日本語に直して伝える。

1. **仕様作成フェーズ（constitution → specify → clarify → plan → tasks）は、形式承認を挟まず連続して実行する。** 各フェーズは対応する Spec-Kit スキルを `Skill` ツールで実行する（最新の Spec-Kit は各ワークフローを `.claude/skills/speckit-*/SKILL.md` というスキルで提供。見つからない場合は該当 `SKILL.md` を `Read` して手順を忠実に実行）。

2. **各フェーズ完了直後に必ず「不確定情報チェック」（下記）を実行する。** チェックの結果、ユーザーの意向確認が必要と判断された論点があるときだけ `AskUserQuestion` で確認し、回答を成果物へ反映してから次フェーズへ進む。確認すべき論点が無ければ、生成物を 1〜2 行で日本語報告してすぐ次へ進む。

3. **実装に入る前に一度だけ、まとめて承認を取る**（下記「実装前ゲート」）。

4. 実装中の破壊的・不可逆な操作の前にはユーザーへ確認する。

## 不確定情報チェック（サブエージェント）

各フェーズ完了後に、**`Task` ツールで読み取り専用のサブエージェント（`Explore` 推奨）を起動**し、直近フェーズの成果物に「ユーザーの意向確認が必要な不確定情報」が無いかを点検させる。

サブエージェントへの指示（要旨）:
> 直近の **<フェーズ名>** フェーズで生成・更新された成果物（例: `<対象ファイルパス>`）をレビューし、**ユーザーの意向確認が必要な不確定情報だけ**を抽出せよ。
> 抽出基準（**両方**を満たすものだけ）:
> - (a) エージェントが妥当に推測・決定できない（ユーザーの好み・目的・制約に依存する）
> - (b) その選択が**仕様・設計・スコープを実質的に左右する**
> 次は**挙げない**: 妥当なデフォルトで決められるもの、些末・様式的なもの、後段フェーズで自然に決まるもの（その場合は前提として記録し確認しない）。
> 重要度の高い順に**最大 4 件**まで。
> 出力は次の JSON のみ:
> `{"needs_confirmation": <bool>, "items": [{"topic": "<論点>", "why": "<なぜ人の意向が必要か>", "options": ["<候補1>", "<候補2>", ...], "recommended": "<推奨候補。あれば>"}]}`
> 確認が不要なら `{"needs_confirmation": false, "items": []}` を返す。

オーケストレーター側の処理:
- `needs_confirmation` が `false` → 何も聞かずに次へ進む。
- `true` → `items` を `AskUserQuestion` にマッピングして確認する（1 回の呼び出しにつき最大 4 問・各 2〜4 択、`recommended` があれば先頭に置き「（推奨）」を付す。選択肢は日本語で）。回答を成果物へ反映（必要なら該当スキルを再実行、または該当ファイルを編集）してから次フェーズへ進む。
- 確認した論点・決定はメモしておき、最後の実装前ゲートの説明にも反映する。

## 実行手順

### 0. 入力の確認
- `$ARGUMENTS` に「作りたいもの」の概要があればそれを起点にする。空なら「何を・なぜ作るのか」を 1〜2 問だけ確認してから開始する。
- `TodoWrite` で全フェーズのチェックリストを作成し進捗を可視化する。
- `.claude/skills/` に `speckit-*` スキルが存在することを確認する。無ければ Spec-Kit が未初期化なので、その旨を伝えて停止する。

### 1〜5. 仕様作成（各フェーズ後に不確定情報チェック）
以下を順に実行する。**各フェーズの直後に必ず「不確定情報チェック」を実行**し、必要なときだけ確認する。

1. **Constitution** — `Skill` で `speckit-constitution`（成果物: `.specify/memory/constitution.md`）→ チェック
2. **Specify** — `Skill` で `speckit-specify`（入力概要を渡す）→ チェック
3. **Clarify** — `Skill` で `speckit-clarify`（このスキル自体の質問も `AskUserQuestion` で行う）→ チェック
4. **Plan** — `Skill` で `speckit-plan` → チェック
5. **Tasks** — `Skill` で `speckit-tasks` → チェック

> 任意: 品質を高めたい場合は `speckit-analyze`（整合性チェック）や `speckit-checklist`（品質チェックリスト）をここで挟んでよい。

### 6. 実装前ゲート（ここで一度だけ承認を取る）
仕様作成が完了したら、実装に入る前に **これまでの成果物を日本語でまとめて説明** する。最低限:
- **何を作るか**（仕様の概要 / 主要要件）
- **どう作るか**（実装計画の要点・主要な技術選定）
- **作業内容**（タスク一覧の要約。`specs/.../tasks.md` などのパスも示す）
- これまでの不確定情報チェックで**確認・決定した事項**、および残る前提・リスク・未確定事項

そのうえで `AskUserQuestion` を 1 回呼び、承認を得る:
- **質問**: 「仕様の作成が完了しました。上記の内容で実装に進んでよいですか？」
- **選択肢**:
  1. `実装を開始` — この内容で `speckit-implement` を実行する。
  2. `仕様を修正` — どこを直したいか聞き、該当フェーズをやり直してから、再度この説明と承認を行う。
  3. `中断する` — ここで停止し、現在地を要約する。

### 7. Implement（承認後）
- `Skill` ツールで **`speckit-implement`** を実行する。
- 実装は段階的に行い、破壊的・不可逆な操作の前にはユーザーへ確認する。
- 完了後 → 最終サマリ（実装したファイル、残課題、次の一手）を日本語で提示して終了する。

## 注意
- 各フェーズの具体的な手順・出力形式は **必ず該当の speckit スキルの記述に従う**（このファイルはあくまで進行役）。
- 不確定情報チェックは**過剰に確認しないこと**が重要。基準 (a)(b) を両方満たす論点だけに絞り、それ以外は前提として記録して進める。
- Spec-Kit のスクリプト（`.specify/scripts/`）が前提の手順は尊重して実行する。
- スキル名・配置が将来変わった場合は、`.claude/skills/` 配下を `Glob` で探索して `speckit-*` を特定すること。
