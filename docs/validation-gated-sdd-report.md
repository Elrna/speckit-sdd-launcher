# SDDワークフロー改修レポート — Validation-Gated SDD+TDD

**対象**: `sdd-launcher/src-tauri/templates/SDD.md`（`/SDD` スラッシュコマンドのテンプレート）
**日付**: 2026-07-02
**ステータス**: ✅ 実装済み（v0.4.0, 2026-07-03）— 本レポートの G1〜G4・トレーサビリティタグ・差し戻し規則は `templates/SDD.md` に、テストの設計原則は `templates/coding-policy.md` に反映済み。あわせて TDD サイクル（Red → G3 → Green → Refactor）の明文化と、`specs/` を用いた中断からの再開判定も追加した。

**v0.4.0 での追加改修**:
- **機械的執行レイヤー（hooks）**: G3 の機械化可能な 2 項（テストの捏造・検証漏れ）を散文指示から `.claude/hooks/trace-lint.ps1` に委譲し、Claude Code の hooks（`.claude/settings.json`）で自動発火させた。**PostToolUse** フックはテストファイル編集時に `SPEC:` 注釈の欠落を警告し、**Stop** フックは応答終了のたびに trace-lint と `.specify/memory/test-command.txt` のテストスイートを実行して、失敗があれば停止をブロックする（テストが失敗したまま作業を終えられない、を機械的に強制）。`specs/` が無い作業では静かに素通りする。G3 の非機械項（期待値が不自然になる仕様）はサブエージェント側の指示として残す。
- **implement のタスク単位サブエージェント化**: 単一コンテキストで全タスクを回すとコンテキスト劣化で指示が抜け落ちるため、`tasks.md` の各タスクを Task ツールの新規サブエージェントで実装する方式に変更した。サブエージェントには該当タスク・関連仕様・具体例・coding-policy のみを渡し（仕様作成過程の経緯は渡さず独立視点を確保）、構造化 JSON（完了/差し戻し提案/残課題）を返させ、差し戻し提案はオーケストレーターが `AskUserQuestion` でユーザーに取り次ぐ。
- SDD Launcher は上記 hooks スクリプト・`settings.json` をテンプレートとして埋め込み、`run_setup` で対象プロジェクトへ配置する。既存 `settings.json` は JSON として安全にマージ（未登録時のみ追記・パース不能なら上書きせずスキップ）する。

**v0.5.0 での追加改修（PonyTail 統合）**:
- **YAGNI ヒューリスティックの移植**: OSS の AI エージェント向けルールセット「PonyTail」（DietrichGebert, 2026-06, MIT — "The best code is the code you never wrote"）の考え方を、新しいゲートやモードを増やさずに既存構造へ織り込んだ。実装層は `templates/coding-policy.md` に「§0 書く前の判断 — YAGNI 決定ラダー」（7 段の決定ラダー・Golden Rules・`ponytail:` コメント規約・セーフティガード）を新設し、TDD の Green/Refactor 指示からこれを参照させる。仕様層は G1 の敵対的レビュー観点2 に過剰仕様（スコープ膨張）の削減提案を統合した（変な仕様と同様、過剰仕様も `[INFERRED]` に潜むため）。hooks は無変更 — trace-lint は「仕様を通さない黙ったスコープ削減」を検証漏れとして検出するため、YAGNI の暴走を止める安全弁として補完関係にある。詳細は第 6 章。

## 1. 現行SDDワークフローの思想

現行の `/SDD` コマンドは、Spec-Kit のフロー（constitution → specify → clarify → plan → tasks → implement)を一括実行するオーケストレーターであり、次の設計思想に基づいている。

### 1.1 自動進行と最小限の確認

仕様作成フェーズは形式的な承認を挟まず連続実行する。「次に進んでいいですか？」という儀式的な確認はユーザーの時間を奪うだけで品質に寄与しない、という判断である。確認は「本当に人の意向が要る論点」だけに絞る。

### 1.2 不確定情報チェックによる選択的な質問

各フェーズ完了後に読み取り専用サブエージェントが成果物を点検し、
(a) エージェントが妥当に推測できず、かつ (b) 仕様・設計・スコープを実質的に左右する
論点だけを抽出して `AskUserQuestion` で確認する。それ以外は妥当なデフォルトで進める。

### 1.3 実装前の一括承認

仕様が完成した時点で一度だけ、成果物をまとめて説明し承認を取る。承認後は `speckit-implement` が仕様に忠実に実装する。TDDと組み合わせる場合、テストは仕様から導出され、実装はテストを通すことで仕様への準拠を保証する。

この思想は「人間の介入コストを最小化しつつ、要所で意向を反映する」点で合理的であり、改修後も維持する。

## 2. 構造的欠陥

現行フローには、思想の良し悪しとは独立した**構造上の欠陥**が4つある。いずれも「仕様が正しい」という暗黙の前提に依存していることに起因する。

### 2.1 validation層の不在 — TDDは仕様の誤りを増幅する

品質保証には2つの独立した問いがある。

| 問い | 名称 | 現行フローでの担当 |
|---|---|---|
| 仕様通りに作れたか | **verification** | TDD（テストが保証） |
| 作るべきものを仕様化できたか | **validation** | **担当が存在しない** |

TDDは仕様への忠実さを保証する装置である。仕様が正しければ強力に働くが、仕様が間違っていれば**その間違いをテストで固定化し、完璧に実装する**。つまりTDDは仕様の品質に対して中立ではなく、誤りの増幅器として作用する。誤った仕様から生成されたテストは全てグリーンになるため、パイプライン内のどの計器も異常を示さない。

### 2.2 自己判定 — 仕様を書いた文脈が仕様の完了を判定している

specify・clarify・plan を実行したのと同じ会話文脈が「仕様は完成した」と判断している。エージェントが要求を誤解していた場合、その誤解は仕様にもテストにも自己評価にも一貫して現れるため、**同一文脈内からは原理的に検出できない**。既存の不確定情報チェックはサブエージェントを使うものの、点検対象は「不足している情報」であり「誤っている情報」ではない。すなわち:

- 不確定情報チェック = **不足の検出**（書かれていないことを見つける）
- 欠けているもの = **誤りの検出**（書かれていることが要求とずれていないか検証する）

### 2.3 要約ベースの承認 — 人間のレビューが機能しない形式で承認を取っている

実装前ゲートでは仕様の「要約」を提示して承認を得る。しかし要約は変な仕様の変さを丸めてしまう。「残高管理機能を実装します」という要約は承認されるが、その裏の「負の残高を許容する」という仕様は目に触れない。人間は抽象的な仕様の散文から欠陥を見抜くのが苦手であり、**具体的な入出力例を見たときに初めて「そうじゃない」と言える**。現行ゲートは人間のレビュー能力が最も働かない形式で唯一の承認を取っている。

### 2.4 仕様の聖典化 — 下流の違和感を上流に還流する経路がない

implement フェーズの指示は「仕様に忠実であること」を要求する。実装エージェントが「仕様通りだが明らかに変だ」と気づいても、忠実さの定義が「仕様に従うこと」である限り、違和感は握りつぶされ変な仕様が忠実に実装される。仕様の欠陥に最も気づきやすいのは、それを具体化する下流工程（テスト作成・実装）であるにもかかわらず、そこからのフィードバック経路が存在しない。

## 3. 改修フロー: Validation-Gated SDD+TDD

### 3.1 設計原則

改修は3つの原則に基づく。いずれも第2章の欠陥への直接の対抗である。

1. **生成と批判の分離** — 仕様を書いた文脈に仕様を検証させない。元の要求だけを持たせた独立サブエージェントに批判させる。（→ 欠陥 2.2）
2. **人間には抽象仕様ではなく具体例を見せる** — 承認ゲートの提示物を要約から具体的な入出力例に変える。（→ 欠陥 2.3）
3. **仕様は聖典ではなく反証可能な仮説** — 下流工程からの差し戻し経路を常設する。（→ 欠陥 2.4）

3原則が揃うことで、欠陥 2.1（validation層の不在）が多層防御として埋まる。

### 3.2 パイプライン全体像

```
要求 ──→ 仕様化（constitution → specify → clarify → plan → tasks）
              │  各要件にトレーサビリティタグ [USER]/[DECIDED]/[INFERRED]
              ▼
        [G1] 仕様妥当性ゲート
              独立サブエージェントによる敵対的レビュー（clarify後）
              ▼
        具体例化（Specification by Example）
              ▼
        [G2] 実装前ゲート
              人間が「具体例」と「[INFERRED]一覧」を見て承認
              ▼
        TDD実装: Red → [G3] テスト・トレースチェック → Green/Refactor
              │    ※全工程共通の「差し戻し規則」
              ▼
        [G4] 要求直結の受け入れ検証（仕様を経由しない最終validation・任意）
```

### 3.3 各要素の役割

#### トレーサビリティタグ

仕様書の各要件に出典タグを付ける: `[USER]`（ユーザーの発言に遡れる）/ `[DECIDED]`（確認済みの決定に遡れる）/ `[INFERRED]`（エージェントの推測・補完）。変な仕様の大半はエージェントが善意で補完した `[INFERRED]` に潜むため、以降のゲートで `[INFERRED]` を重点的に晒す。検証コストを疑わしい箇所に集中させるための基盤である。

#### G1: 仕様妥当性ゲート（新設・最重要）

clarify 完了後、**ユーザーの元の要求文と仕様書だけ**を入力とする独立サブエージェントを起動する。仕様作成過程の文脈を渡さないことで、作成時の誤解や思い込みから自由な視点を確保する。レビュー観点は3つ:

- **意図との乖離** — この仕様を完璧に実装したとき、元の要求者は満足するか
- **出所不明の要件** — どの `[USER]`/`[DECIDED]` にも遡れない要件の妥当性（捏造・過剰仕様の疑い）
- **矛盾・非常識** — 要件間の矛盾、ドメイン常識に反する挙動

既存の不確定情報チェック（不足の検出）とは役割が異なり、両者は併存する。

#### G2: 実装前ゲートの置換強化

主要要件ごとに **Given/When/Then + 具体的な値の入出力例** を生成し、承認ゲートの提示物を「仕様の要約」から「具体例一覧 + `[INFERRED]` 要件一覧」に変える。具体例は承認後に破棄せず、implement フェーズで受け入れテストの雛形として再利用するため、追加コストはほぼ相殺される。

#### G3: テスト・トレースチェック

TDDのRed段階で、各テストに検証対象の仕様項目を対応させる。これにより3種類の異常が機械的に検出できる:

- どの仕様にも対応しないテスト → **テストの捏造**
- どのテストにも対応しない主要仕様 → **検証漏れ**
- テストに書き下すと期待値が明らかに不自然になる仕様 → **変な仕様の最終検出網**（差し戻し規則へ）

3点目が重要である。仕様の散文では気づかれなかった変さは、`assert balance == -500` のような具体的なアサーションに変換された瞬間に露呈することが多い。

#### 差し戻し規則（全フェーズ共通の恒常原則）

「仕様通りだが変だと感じたら、実装せず停止し、仕様変更提案（該当仕様項目・違和感の内容・代案）をユーザーに提示する。**仕様の欠陥報告は仕様への忠実さより優先する**。」

これは忠実さの定義の書き換えである。SDDの弱点は「仕様に従うこと」が正しさの定義になる点にあり、この規則によって仕様は「従うべき聖典」から「反証可能な仮説」に格下げされる。

#### G4: 要求直結の受け入れ検証（任意）

実装完了後、**元の要求文だけ**を渡した独立サブエージェントが成果物を実際に確認し、仕様を経由せず「要求は満たされたか」を判定する。仕様経由のテストが全てグリーンでも、仕様自体が要求からずれていればここで落ちる。実行コストがあるため品質重視の場合の任意ステップとする。

### 3.4 欠陥と対策の対応

| 構造的欠陥 | 対策 | 機構 |
|---|---|---|
| 2.1 validation層の不在 | G1 + G4 | 要求と成果物を直接突き合わせる検証を仕様の前後に配置 |
| 2.2 自己判定 | G1（+G4）の入力制限 | 元の要求のみを持つ独立文脈に批判させる |
| 2.3 要約ベースの承認 | G2 | 人間のレビュー能力が働く「具体例」形式で承認を取る |
| 2.4 仕様の聖典化 | 差し戻し規則 + G3 | 下流の違和感を仕様変更提案として上流へ還流 |

多層になっているのは意図的である。単一のゲートは（それ自体もLLMによる判定である以上）見逃しうるが、**独立した視点（G1）・人間の直観（G2）・具体化による露呈（G3）・実物での確認（G4）は失敗モードが異なる**ため、同じ変な仕様が全層をすり抜ける確率は大きく下がる。

## 4. 思想の継承とコスト管理

改修は現行思想を否定するものではない。「儀式的な確認はしない」「確認は人の意向が要る論点だけ」「承認は実装前の一度だけ」はすべて維持される。追加されるのは人間への確認ではなく、**エージェント間の検証**であり、人間に届くのは検証をすり抜けられなかった論点だけである。

コストの観点では、G1・G2・G3・差し戻し規則をコア、G4を任意と位置づける:

| 要素 | 追加コスト | 効果 |
|---|---|---|
| G1 + トレーサビリティタグ | サブエージェント1回 | 問題の根本（要求→仕様のずれ）を直接検出 |
| G2 具体例化 | ほぼゼロ（受け入れテスト雛形として再利用） | 人間レビューの精度が大幅に向上 |
| G3 + 差し戻し規則 | プロンプト追記のみ | 最終検出網・還流経路の常設 |
| G4 | サブエージェント1回 + 動作確認 | 品質重視プロジェクト向けの最終保険 |

## 5. 残存リスクと限界

改修後も残る限界を明記しておく。

- **G1も LLM である** — 独立文脈による批判は誤解の相関を切るが、モデル共通のバイアス（もっともらしい仕様を妥当と判断する傾向）までは切れない。最終防衛線はあくまで G2 の人間承認であり、ユーザーが具体例を実際に読むことが前提となる。
- **要求そのものが誤っている場合は救えない** — 本フレームワークは「要求→仕様」のずれを検出するものであり、要求自体の妥当性（そもそもそれを作るべきか）は対象外である。
- **具体例の網羅性** — G2 で提示する具体例は主要要件に限られる。エッジケースの変な仕様は G3（テスト化時の露呈）と差し戻し規則が受け持つが、保証ではない。
- **差し戻しの感度調整** — 差し戻し規則が過敏だと実装が頻繁に停止し、現行思想（確認の最小化）と衝突する。「明らかに変」の閾値は運用の中で調整が必要になる。

これらは欠陥ではなくトレードオフであり、完全自動での validation は原理的に不可能（要求の意図は要求者の頭の中にしかない）である以上、**人間の判断力が最も働く形式・タイミングに検証を集約する**のが本改修の到達点である。

## 6. PonyTail 統合 — YAGNI の移植（v0.5.0）

### 6.1 統合方針

PonyTail は工程論ではない。「コードを書く直前に、何を・どれだけ書くかをどう判断するか」という意思決定ヒューリスティックである。これは本フローが第3章で組んだ G1〜G4 の縦の骨格（validation/verification の各ゲート）とは直交する。骨格が「仕様と実装が正しいか」を検証する層だとすれば、PonyTail は各ゲートの内側で「そもそもこれを書く必要があるか」を判断する横の層であり、置換ではなく上乗せである。したがって新しいゲートやモードを追加せず、既存の構造物（coding-policy・G1・実装サブエージェント指示・臭いチェックリスト）に部品として埋め込んだ。

### 6.2 移植先の対応表

| PonyTail の要素 | 移植先 | 役割 |
|---|---|---|
| 決定ラダー・Golden Rules・`ponytail:` コメント規約・セーフティガード | `coding-policy.md` §0（新設） | 実装時の判断基準の本体 |
| 過剰仕様（スコープ膨張）の削減提案 | G1 敵対的レビュー観点2 | 仕様段階での YAGNI 検出 |
| ラダーの参照指示 | 実装サブエージェント指示（Green・Refactor） | 実装時に §0 を参照させる導線 |
| YAGNI 違反の項目 | 臭いチェックリスト項目11 | 実装レビュー時の点検観点 |

G1〜G4・差し戻し規則・hooks（trace-lint・PostToolUse/Stop）は無変更である。PonyTail は既存の検証網の中を流れる「判断材料」であって、新しい検証網ではない。

### 6.3 なぜ仕様層にも適用するか

本フローの中核知見は「変な仕様の大半は `[INFERRED]` に潜む」であった（第3.3節）。これは過剰仕様にもそのまま当てはまる。スコープ膨張は多くの場合、エージェントが要求の行間を善意で埋める仕様作成段階で発生する。YAGNI を実装層だけに置くと、膨らんだ仕様そのものは素通りし、実装エージェントはその仕様に忠実であろうとして結局は過剰実装を行う。忠実さの徳目が YAGNI の判断を上書きしてしまう。

G1 は「ユーザーの元の要求文と仕様書だけ」を持つ独立文脈であり、要求に遡れない項目を機械的に洗い出せる立場にある。要求に無い項目はほぼ過剰仕様の候補であるため、G1 は YAGNI 検出の構造上最適な位置にある。実装層からの仕様削減提案（差し戻し規則経由）はあくまで最終防衛線であり、主担当は G1 に置く。

### 6.4 衝突分析と解決

PonyTail の「最小限に書く」思想は、既存フローの一部の原則と字面上衝突する。衝突点ごとに調停方針を定めた。

| 衝突 | 解決 |
|---|---|
| 最小実装 vs 境界値・エラー系テスト必須 | セーフティガードにテスト網羅性を明記する。最小化の対象は**解の構造**であって**テストの網羅性**ではない、と適用範囲を分離した |
| 仕様要件の削減 vs trace-lint の検証漏れ検出 | 要件削除は必ず spec.md からの削除（G1 での提案 → 差し戻し規則 → ユーザー承認）を伴わせる。trace-lint は「仕様を通さない黙ったスコープ削減」を禁じる番人として機能し、YAGNI の暴走を止める安全弁になる |
| YAGNI 提案の乱発 vs 確認の最小化思想（第1.1節） | 判断の粒度を分離した。実装の簡素化（how）はサブエージェントの裁量に委ね確認不要とし、仕様の削減（what）は G1 が主担当とし、実装層からの削減提案は例外的な検出網に留める |
| 抽象化禁止 vs DRY・ポリモーフィズム推奨 | 「実装が2つ現れてから抽象化する」という時間軸の条件で調停する。最初の1実装では禁止、重複が実際に生じた時点で許可という順序を Golden Rules に明記した |

### 6.5 見送った案（意図的な不採用 — 「作らない」の実践）

PonyTail 自体の理念（書かないコードが最良のコード）を統合作業そのものにも適用し、次の案は検討のうえ採用しなかった。

- **強度モード（lite/full/ultra/off）**: 儀式的な設定ノブは第1.1節の「確認の最小化」思想と衝突する。モードを増やす代わりに full 相当の内容を唯一の既定として固定した。ultra が想定する「要求そのものへの異議」は既存の G1＋差し戻し規則が代替し、lite が想定する「軽量な確認」は既存の不確定情報チェック A が代替するため、いずれも新設の必要がなかった。
- **speckit スキル本体への注入・専用ゲートの新設・`ponytail:` 強制フック・技術的負債の収集コマンド**: いずれも新しい構造物を作る案であり、「最小の変更で最大の効果を狙う」という PonyTail 自身の理念と矛盾する。既存の骨格（coding-policy・G1・臭いチェックリスト）へ部品として埋め込む方針で足りるため、見送った。
