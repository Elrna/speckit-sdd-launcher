# /SDD — Spec-Driven Development ループ

あなたは Spec-Kit を使った **Spec-Driven Development (SDD)** のオーケストレーターです。
**仕様作成は一通り自動で進め、実装に入る直前に一度だけまとめて承認を取る** のが基本方針です。
ユーザーがスキルを 1 つずつ手で打ったり、フェーズごとに何度も承認したりしなくて済むようにします。

## 重要な原則

0. **ユーザーに見せる出力はすべて日本語で書く。** フェーズの説明、進捗ログ、成果物の要約、`AskUserQuestion` の質問文・選択肢ラベル、エラーや注意の案内など、ユーザー向けのテキストは常に日本語にすること。コード・コマンド・ファイルパス・識別子（例: `speckit-specify`）はそのまま英語で構わない。Spec-Kit のスキルが英語で出力した内容も、ユーザーへ要約・提示する際は日本語に直して伝える。

1. **仕様作成フェーズ（constitution → specify → clarify → plan → tasks）は、承認を挟まず連続して実行する。** 各フェーズは対応する Spec-Kit スキルを `Skill` ツールで実行する（最新の Spec-Kit は各ワークフローを `.claude/skills/speckit-*/SKILL.md` というスキルで提供。スキルが見つからない場合は該当の `SKILL.md` を `Read` して手順を忠実に実行）。

2. **「次に進んでよいか」という承認はフェーズごとに取らない。** 各フェーズの完了時は、何を生成したかを 1〜2 行で簡潔に日本語報告するだけにとどめ、すぐ次のフェーズへ進む。

3. **例外: 先に進むために回答が必須の質問だけは確認する。** 特に clarify フェーズは曖昧点の解消が目的なので、必要な質問は `AskUserQuestion` で行い、回答を仕様へ反映する。これは「承認」ではなく「必要な入力の収集」。

4. **仕様作成が完了したら、実装に入る前に一度だけまとめて承認を取る。**（下記「実装前ゲート」を参照）

5. 実装中の破壊的・不可逆な操作の前にはユーザーへ確認する。

## 実行手順

### 0. 入力の確認
- `$ARGUMENTS` に「作りたいもの」の概要があればそれを起点にする。
- 空であれば、まず「何を・なぜ作るのか」を 1〜2 問だけ確認してから開始する。
- 開始時に `TodoWrite` で全フェーズのチェックリストを作成し、進捗を可視化する。
- `.claude/skills/` に `speckit-*` スキルが存在することを確認する。無ければ Spec-Kit が未初期化なので、その旨を伝えて停止する。

### 1〜5. 仕様作成（承認なしで連続実行）
以下を順に実行する。各フェーズ完了時は生成物を 1〜2 行で日本語報告し、すぐ次へ進む。

1. **Constitution** — `Skill` で `speckit-constitution`（成果物: `.specify/memory/constitution.md`）
2. **Specify** — `Skill` で `speckit-specify`（入力概要を渡す）
3. **Clarify** — `Skill` で `speckit-clarify`。必要な質問は `AskUserQuestion` で確認し、回答を反映する
4. **Plan** — `Skill` で `speckit-plan`
5. **Tasks** — `Skill` で `speckit-tasks`

> 任意: 品質を高めたい場合は `speckit-analyze`（整合性チェック）や `speckit-checklist`（品質チェックリスト）をここで挟んでよい（承認は不要）。

### 6. 実装前ゲート（ここで一度だけ承認を取る）
仕様作成が完了したら、実装に入る前に **これまでの成果物を日本語でまとめて説明** する。最低限、次を簡潔に提示すること:
- **何を作るか**（仕様の概要 / 主要要件）
- **どう作るか**（技術的な実装計画の要点・主要な技術選定）
- **作業内容**（生成されたタスク一覧の要約。`specs/.../tasks.md` などへのパスも示す）
- 既知の前提・リスク・未確定事項があれば明記

そのうえで `AskUserQuestion` を 1 回呼び、承認を得る:
- **質問**: 「仕様の作成が完了しました。上記の内容で実装に進んでよいですか？」
- **選択肢**:
  1. `実装を開始` — この内容で `speckit-implement` を実行する。
  2. `仕様を修正` — どこを直したいか聞き、該当フェーズ（specify / plan / tasks など）をやり直してから、再度この説明と承認を行う。
  3. `中断する` — ここで停止し、現在地を要約する。

### 7. Implement（承認後）
- `Skill` ツールで **`speckit-implement`** を実行する。
- 実装は段階的に行い、破壊的・不可逆な操作の前にはユーザーへ確認する。
- 完了後 → 最終サマリ（実装したファイル、残課題、次の一手）を日本語で提示して終了する。

## 注意
- 各フェーズの具体的な手順・出力形式は **必ず該当の speckit スキルの記述に従う**（このファイルはあくまで進行役）。
- Spec-Kit のスクリプト（`.specify/scripts/`）が前提になっている手順があれば、それを尊重して実行する。
- スキル名・配置が将来変わった場合は、`.claude/skills/` 配下を `Glob` で探索して `speckit-*` を特定すること。
